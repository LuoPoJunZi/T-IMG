## 风险与控制

- 已跟踪依赖会制造巨大安装差异：已从 Git 索引取消跟踪并由 `.gitignore` 排除，本地依赖目录仍保留；大规模删除差异需在提交前重点审查。
- `type: module` 会影响 Node 对 `.js` 的解析：现有 Functions 本身已使用 ESM，验证 `npm test` 和 Wrangler 编译以控制风险。
- 不采用外部代码同步：项目依靠独立维护、依赖审计和完整测试控制变更风险。
- 缺少真实 Telegram、KV 和审核凭据：不使用生产密钥，本次只验证无外部写入的本地链路。
- 上传认证依赖 Pages Functions：生产项目必须保留 `_routes.json` 并将配额行为设为 Fail closed，防止 Function 不可用时静态页面失败开放。
- 上传认证不保存失败次数：必须使用至少 24 个随机字符的生产访问密码；错误请求仍消耗 Functions 配额，高风险公开部署应在自定义域名叠加 Cloudflare WAF/Rate Limiting、Turnstile 或 Access。
- 自定义域名 WAF 只覆盖该域名区域：必须收口生产 `*.pages.dev` 旁路；规则只质询登录入口，避免公开图片、上传 XHR 和 API 客户端收到 Challenge Page。
- `img_url` 是上传索引必需绑定：上传前校验 KV 方法以避免无索引文件；Telegram 已成功后的瞬时 KV 写入失败只记录安全警告，避免用户重试产生重复文件。
- 自动短链依赖 `img_url` 映射：使用 72 位随机短码，不做额外碰撞读取；短链访问每次一次 KV 读取，不写回未变化元数据。KV 不可用时旧 `/file/` 链接仍可服务，短链返回 503。
- npm 的新公告可能先于开发工具上游修复：生产依赖高危审计继续阻断；开发工具审计保持可见但不阻断测试，禁止为清零报告而强制降级 Wrangler 或覆盖其未验证的 Miniflare/Sharp 依赖。
